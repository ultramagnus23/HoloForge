## The K=7.85 question — resolved: real value, coincidental, not a transcription error

`oracle` PSNR at `period8` = **7.853769666 dB**. `K` at `period8` =
**7.853981634 rad/µm**. These are two different physical quantities
(a PSNR in dB, a spatial frequency in rad/µm) computed by entirely different
code paths (`oracle_ideal`'s PSNR vs. `2π/(period_px·dx)`), that happen to
agree to 4 significant figures (difference: 2.1×10⁻⁴) purely because
`period=8px, dx=0.1µm` gives `K = 2π/0.8 = 2.5π ≈ 7.85398`, and this
particular medium/target/iteration-budget combination happens to also
produce an oracle PSNR near 7.85 dB. **Not a bug, not a copy-paste error —
verified independently-computed values that are numerically close by
chance.** No action needed beyond noting it (worth a one-line footnote in
the manuscript so a sharp-eyed referee doesn't suspect what this audit just
ruled out).

## GS margin range — imprecise, not fabricated

The abstract/prose claims media-in-the-loop beats GS "by 0.90 to 4.33 dB."
Tracing this: **0.90 dB is real** (period8, both seeds: 0.898/0.900).
**4.33 dB is period24 seed=0 specifically** (4.329) — but period24 seed=1
gives **4.546 dB**, which is higher. Unlike `ours`/`blind`/`oracle` (which
were bit-identical across seeds due to the now-fixed bug), `media_blind_gs`
was never part of that bug — its randomness came from whatever the global
RNG state happened to be after prior calls in the same script, so it
genuinely does vary between the "seed=0" and "seed=1" rows (its seed
parameter was itself only added a proper meaning this session). The claimed
range **understates the true seed-0/seed-1 range**, which is [0.898, 4.546],
not [0.90, 4.33]. Minor, but worth a one-word fix ("4.33" → "4.55" or
"up to ~4.5") before this number is trusted in a submission — flagged here
rather than silently corrected, since ground rule 4 says the narrative
follows the data, and this is exactly the kind of small drift Phase 3's
proper multi-seed GS logging (Phase 2's M1 registry entry) will settle for
good.

`part2_media_draft_v1.md` (not the tex) quotes GS=6.54, blind=9.04,
ours=16.64, oracle=26.57 dB from `experiments/f2_panel.py`. This script calls
`media_in_the_loop`/`media_blind_sgd`/`oracle_ideal` at their **default**

including Phase 5's F3 reconstruction panels which reuse this same

## 3D showcase (+0.63 dB) — separate, still-live copy of the same bug class, but not currently a false claim

`experiments/showcase_3d.py` defines its own local `media_in_the_loop_3d`
with `theta = torch.zeros(rec.n_x, rec.n_y, ...)` — a second, independent
copy of the exact zero-init pattern that was fixed in `holomedia/optimize.py`
this session, **not itself fixed**, because it's a standalone function in a
different file. This is currently **not a false claim**: the tex's 3D
showcase paragraph reports a single run with no seed-independence claim
attached to it, so the bug is latent, not actively misleading anyone right
now. But if Phase 3/Phase 5 ever want a multi-seed 3D result, this function
needs the same fix (or should be refactored to call
`holomedia.optimize.media_in_the_loop` on a flattened `(n_x*n_y,)` view
instead of duplicating the optimizer). **Flagged for Phase 1/2, not fixed in
this audit pass** — Phase 0's mandate was verification, not new fixes beyond
what's already covered by `test_seed_independence.py`.

## Gradient-pathway ablation — zero-init is correct here, not a bug

`experiments/ablation_gradients.py`'s `optimize_with_grad_fn` also uses
`theta = torch.zeros(...)`, but this is **methodologically correct, not a
bug**: the script's entire point is comparing three gradient *pathways*
(unrolled / checkpointed-adjoint / neural-surrogate) starting from an
*identical* initial condition, so that differences in the downstream
optimization are attributable to the gradient source, not to different
random starting points. A shared deterministic zero-init is the right
control here. No action needed.
